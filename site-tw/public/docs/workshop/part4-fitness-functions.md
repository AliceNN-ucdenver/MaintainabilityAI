## What is a fitness function

The term comes from Neal Ford and Rebecca Parsons's *Building Evolutionary Architectures*. A **fitness function** is an automated check that some architectural characteristic continues to hold as the codebase evolves. Not a test (which checks behaviour). Not a lint rule (which checks style). A **structural assertion about the codebase as a whole**.

Examples:

- "No function in this repo has cyclomatic complexity above 10."
- "There are no circular imports between modules."
- "Code in the route layer does not import directly from the database layer; it goes through the service layer."
- "No dependency in `package.json` is older than 180 days."

A fitness function fails the build when the characteristic stops holding. It is the architectural equivalent of a unit test: cheap to run, mechanical to interpret, and devastating when ignored.

### The five we generate today

| Fitness function | Tool | Threshold | Blocking? |
|---|---|---|---|
| **Complexity** | ESLint `complexity` rule or `pmat` | ≤ 10 per function | **Yes** |
| **Duplicated code** | `jscpd` | ≤ 3% across `src/` | No (advisory) |
| **Dead code** | `knip` or `ts-prune` | Zero unused exports | No (advisory) |
| **Import boundaries** | `madge` | Zero circular imports | No (advisory) |
| **CALM layer enforcement** | Custom AST walker + CALM model | Zero violations of declared layer rules | **Yes** |

Two are blocking, three are advisory. The reasoning is in the Q&A below. The instinct that "everything blocking is best" is wrong. A noisy gate teaches the team to bypass gates.

## The agentic shift, applied to gates

Part 3 taught you the Cheshire enrich → assign → review loop on a security finding. Part 4 applies the *same loop* to a different problem class: building the gates themselves.

You will not write any of the five fitness functions by hand today. You will write an RCTRO for "add five fitness functions to this repo", let Cheshire pull the maintainability packs in, assign the agent, and review the PR that comes back. The agent writes the jscpd config, the knip config, the madge invocation, the GitHub Actions workflow that runs them all on every PR.

The one piece you co-author with the agent is the CALM layer enforcement pack. That one does not exist in the shipped prompt library because *every BAR's CALM model is different*. You write the rules; the agent writes the checker. This is also where your team starts building its own packs.

### Step 2. Co-author the CALM layer enforcement pack

Open a new file at `.cheshire/prompts/maintainability/calm-layer-enforcement.md`. Paste this skeleton.

```markdown
---
name: CALM Layer Enforcement
version: 0.1.0
applies-to: Any BAR with a declared CALM model
---

## Role
Senior architect writing a structural assertion that derives layer rules
from the BAR's CALM model and fails the build when those rules are violated.

## Context
- The BAR's CALM model declares nodes (components) and flows (allowed calls between them).
- Code in this repo maps to one or more nodes via convention: `src/routes/*` is the route layer,
  `src/services/*` is the service layer, `src/db/*` and `src/models/*` are the data layer.
- A violation is any import statement that crosses a layer boundary not declared in the CALM flow.

## Task
Write a Node script (`scripts/check-calm-layers.cjs`) that:
1. Reads the BAR's CALM model from the path declared in `.cheshire/config.json`.
2. Walks `src/` with a TypeScript AST parser and collects every import statement.
3. Maps each source file to a CALM node by convention (routes / services / data).
4. Asserts that every import respects the declared layer flow.
5. Emits a SARIF-compatible report on violation with file:line citations.

## Requirements
- Zero false positives on the seed code if no violations exist.
- Exit non-zero on any violation.
- Output: human-readable summary + machine-readable SARIF.
- Configuration: layer-to-directory mapping is in `.cheshire/calm-layers.json` so it can be tuned per BAR.
- Tests: a fixture with a synthetic violation must fail; the real `src/` (after fixes) must pass.

## Output
- `scripts/check-calm-layers.cjs`
- `.cheshire/calm-layers.json` (the route → service → data mapping)
- A test fixture under `scripts/__tests__/calm-layers.fixture/`
- A line in `.github/workflows/fitness-functions.yml` invoking the script
```

This is the first prompt pack your team wrote. It is shorter than the OWASP packs because it is younger. It will grow as you discover edge cases.

### Step 6. Decide what to do with the failures

This is the moment that separates teams who *have* fitness functions from teams who *use* them. The Complexity gate is failing. You have three options.

## The CALM layer function in detail

The fifth fitness function deserves its own paragraph because it is the bridge into the rest of the workshop.

What it does, mechanically: parses every `import` statement in `src/`, maps each file to a layer via the `.cheshire/calm-layers.json` mapping, checks each import against the declared CALM flow, fails if a route imports from `db/` directly (skipping the service layer), or if `services/celebrity-service.ts` imports from `db/identity.ts` (cross-BAR data access), or any other layer-crossing not explicitly declared.

Why this matters for Part 7. The Red Queen's PreToolUse hook does the same check, but **before the agent's edit ever lands on disk**. The CI fitness function is *advisory* in the sense that it fires after the code is written. The Red Queen hook is *deterministic* in the sense that it blocks the write before it happens. The function you generated today is the architectural rule. Part 7 turns the rule into a guardrail at agent-tool-call time.

This is why we generate the function now and the hook later. You need to *believe* the rule before you let it block your agent's tool calls.

## Q&amp;A

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 1. Why only two blocking gates? The complexity and CALM layer rules feel small.</summary>

A blocking gate has to clear three bars:

1. **High signal.** When it fails, the failure is real, not noise. Complexity ≤10 and CALM layer violations are both unambiguous: the number is or is not above the threshold; the import is or is not crossing a declared layer.
2. **Cheap to fix.** A complexity violation is a refactor. A layer violation is moving an import through the service layer. Neither is a multi-day archaeology project.
3. **Worth blocking PRs for.** A team that ships a high-complexity function ships a function the next engineer cannot safely change. A team that ships a layer violation has just made the architecture diagram a polite suggestion.

The other three (duplicated code, dead code, circular imports) fail those bars in slightly different ways. Duplicated code at 4% is sometimes the right answer (test fixtures, generated code). Dead code in a feature flag's off-branch is intentional. A circular import in a generated file is acceptable. These gates *should exist*, because the trend matters (the duplication should not creep from 3 to 5 to 8 percent), but blocking on them creates noise that trains the team to ignore gates. Advisory keeps the signal alive without blunting the signal of the blocking gates.

The chief-trainer rule: **block on what you are willing to fix in the PR; advise on what you want to trend over months.**

</details>

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 2. The agent generated the CALM layer checker. How do I trust the checker is correct?</summary>

Three checks, in order.

1. **The fixture test.** The RCTRO required a synthetic violation fixture. Run it. If the fixture passes (no violation detected when one should be), the checker is broken; comment on the PR and regenerate.
2. **The audit on real code.** Run the checker against the current `src/` and read the output. If it flags zero violations and you can see (by eye) a route file importing from `db/`, the layer mapping is wrong. Adjust `.cheshire/calm-layers.json` and re-run.
3. **The reverse audit.** Intentionally introduce a violation in a branch. Confirm the checker catches it. Revert.

This is how you validate any structural check, agent-generated or not. The fitness function for the fitness function is the fixture test plus the manual audit. The cost is one hour the first time. The benefit is years of trustworthy enforcement.

</details>

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 3. What stops the team from disabling a noisy fitness function instead of fixing the underlying issue?</summary>

Two things.

**The disable goes through the same review.** Disabling a gate is a PR. The PR needs a reason. The reason becomes part of the audit trail. "We disabled the complexity gate because too many functions failed" is a much harder conversation than "we deferred one function and tightened the gate next sprint".

**The Red Queen, in Part 7.** Once Red Queen is installed, the policy engine reads the fitness function outcomes and adjusts the tier. A repo whose gates are disabled by override drops a tier. Lower tier means more PreToolUse hook restrictions on the agent. **Disabling gates makes the agent slower, not faster.** That alignment is the whole point of the trust-battery model.

The structural answer: do not rely on willpower. Wire incentives so the lazy path produces governed outcomes.

</details>

**Packs that apply:** `maintainability/fitness-functions` (the meta), `maintainability/single-responsibility` (an `any` is often a missing type definition that hides multiple responsibilities), `maintainability/technical-debt` (justified `any` is acknowledged debt).

## What you learned

- **A fitness function is a structural assertion**, not a unit test and not a lint rule. It says something about the *codebase as a whole* (no functions over CC 10, no circular imports, no layer crossings) and fails the build when that property stops holding.
- **The five canonical functions:** Complexity (`pmat` or ESLint), Duplicated code (`jscpd`), Dead code (`knip`), Import boundaries (`madge`), CALM layer enforcement (custom AST walker that reads the BAR file).
- **The Cheshire enrich → assign → review loop generalises.** Same workflow you ran in Part 3 on a security fix, applied here to build the gates themselves. *The pattern is the leverage*, not any one application of it.
- **Blocking vs advisory is a judgment call, not a default.** Block what you would fix in this PR (Complexity violation, CALM layer crossing). Advise what you want to trend over months (duplication, dead code, circular imports). Block too much and the team learns to bypass gates.
- **Your team co-authored its first prompt pack.** `.cheshire/prompts/maintainability/calm-layer-enforcement.md` did not exist before today. The OWASP and STRIDE packs are starter content; team-specific packs are how you encode *your* architectural rules.
- **The CALM layer function is the bridge to Part 7.** Today it runs in CI (advisory: after the agent wrote the code). In Part 7 the same rule becomes a PreToolUse hook (deterministic: blocks the write before it lands). Same constraint, two enforcement points.
- **Gate failures need a triage decision, not a knee-jerk fix.** Three honest options: fix now (issue + agent run), defer with a tracked issue, or temporarily lower the threshold with a ratchet date. Disabling a gate is the option that should require the most justification.
- **Validating an agent-generated checker is a fitness function for the fitness function.** Run the fixture (synthetic violation), audit the real `src/`, do the reverse audit. One hour of validation buys years of trustworthy enforcement.

## The Golden Rule for this part

> **Rule 4. Measurement enables governance. If you can't measure it, you can't govern it.**
