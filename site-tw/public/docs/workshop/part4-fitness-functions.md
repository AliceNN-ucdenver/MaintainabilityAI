## What is a fitness function

The term comes from Neal Ford and Rebecca Parsons's *Building Evolutionary Architectures*. A **fitness function** is an automated check that some architectural characteristic continues to hold as the codebase evolves. Not a behaviour test. Not a lint rule. A **structural assertion about the codebase as a whole**.

In this framework a fitness function is a **committed test** — `tests/fitness/<category>.test.ts` — that:

- runs a **fast, native** tool over the repo (`jscpd`, `knip`, ESLint's `complexity` rule, `dependency-cruiser`),
- compares the result against a **baseline floor** in `tests/fitness/baselines.json`,
- **fails the build when the metric gets worse than the floor**, and
- ratchets: as you improve, you lower the floor toward a target, locking in the gain.

> **It is a test, not a workflow.** Fitness functions live under `tests/fitness/` and run in the repo's existing CI alongside the unit tests — there is no separate `fitness-functions.yml`. And the runtime tool is always a **fast native** one. **Never** run heavyweight analyzers like `pmat` inside a fitness test — `pmat` is an authoring-time aid you use locally to *design* the test, never a runtime dependency (it compiles and can take many minutes).

### The four structural categories

| Category | Native tool (TypeScript) | What the baseline tracks |
|---|---|---|
| **Complexity** | ESLint `complexity` rule | max cyclomatic complexity per function |
| **Duplicated code** | `jscpd` | % duplicated tokens across `src/` |
| **Dead code** | `knip` | count of unused exports |
| **Architecture** | `dependency-cruiser` (+ CALM-layer rule) | count of import-boundary / layer violations |

The tool per category is chosen on the Scaffold project-config page and recorded in `.github/repo-metadata.yml` (the `fitness:` block), so the Cheshire recipe knows which tool to wire when it generates each test. Python repos get `radon` / `vulture` / `import-linter`; Go gets `gocyclo` / `deadcode` / `depguard`; etc.

## The agentic shift, applied to gates

Part 3 taught you the Cheshire enrich → dispatch → review loop on a security finding. Part 4 applies the *same loop* to a different problem class: building the gates themselves.

You will not hand-write the fitness tests today. You will use Cheshire's **per-category fitness recipe** ("create a fitness test for *X*"), let it pull the maintainability packs in and read the repo's chosen tool from `repo-metadata.yml`, dispatch Alice, and review the PR that comes back. Alice writes the test, wires the native tool, **runs it once to capture the floor**, and writes the baseline.

The one piece you co-author with the agent is the CALM layer rule inside the architecture test. That rule does not exist in the shipped prompt library because *every BAR's CALM model is different*. You write the rules; the agent writes the checker. This is also where your team starts building its own packs.

## Walkthrough: agent-generated fitness tests

### Step 1. Generate the complexity fitness test

In VS Code with the movie-api workspace open, open **Cheshire Cat → Security Scorecard**. In the fitness-tests panel, pick **Complexity** and click **Create fitness test**. Cheshire:

1. reads the chosen complexity tool from `.github/repo-metadata.yml` (ESLint `complexity` for TypeScript),
2. builds an RCTRO from the maintainability packs + the fitness recipe (which says, in bold, *the test must be fast and native — never pmat at runtime*),
3. files the issue and is ready to dispatch.

Dispatch Alice. The plan should say: add `tests/fitness/complexity.test.ts`, run ESLint's `complexity` rule over `src/`, capture the current max as the floor in `tests/fitness/baselines.json`, and fail when any function exceeds the floor. Review, iterate, merge.

Repeat for **Duplicated code** (`jscpd`) and **Dead code** (`knip`). Each is one recipe → one dispatch → one review.

### Step 2. Co-author the CALM layer rule (the architecture test)

The architecture test is the one you co-author. Open a new pack at `.cheshire/prompts/maintainability/calm-layer-enforcement.md`:

```markdown
---
name: CALM Layer Enforcement
version: 0.1.0
applies-to: Any BAR with a declared CALM model
---

## Role
Senior architect writing a structural assertion that derives layer rules
from the BAR's CALM model and fails the build when those rules are violated.

## Context
- The BAR's CALM model declares nodes (components) and flows (allowed calls between them).
- Code maps to nodes by convention: `src/routes/*` is the route layer,
  `src/services/*` is the service layer, `src/db/*` and `src/models/*` are the data layer.
- A violation is any import statement that crosses a layer boundary not declared in the CALM flow.

## Task
Write `tests/fitness/architecture.test.ts` that:
1. Reads the BAR's CALM model from the path declared in `.cheshire/config.json`.
2. Runs dependency-cruiser (fast, native) over `src/` and collects cross-layer imports.
3. Maps each source file to a CALM node by convention (routes / services / data).
4. Asserts every import respects the declared layer flow, against the baseline floor.

## Requirements
- Fast + native (dependency-cruiser). Never pmat at runtime.
- Zero false positives on the seed code if no violations exist.
- The test fails when the violation count exceeds the baseline floor.
- Configuration: layer-to-directory mapping in `.cheshire/calm-layers.json`, tunable per BAR.
- A fixture with a synthetic violation must fail; the real `src/` (after fixes) must pass.

## Output
- `tests/fitness/architecture.test.ts`
- `.cheshire/calm-layers.json` (the route → service → data mapping)
- A test fixture under `tests/fitness/__fixtures__/calm-layers/`
- the floor entry in `tests/fitness/baselines.json`
```

This is the first prompt pack your team wrote. It is shorter than the OWASP packs because it is younger. It will grow as you discover edge cases.

### Step 3. Generate the architecture test with the new pack

Back in the Scorecard, pick **Architecture** → **Create fitness test**. Cheshire now sees the brand-new calm-layer pack and produces a denser RCTRO. Dispatch Alice. Make sure the plan acknowledges the CALM layer pack — if it glosses over it ("will add a custom layer check"), say so on the PR and re-dispatch: *the architecture test is not generic; use the calm-layer-enforcement pack and the layer mapping it specifies.*

### Step 4. Review the PRs

Each fitness test PR should add roughly:

```
tests/fitness/<category>.test.ts             (the test)
tests/fitness/baselines.json                 (floor + target per category)
package.json                                 (jscpd / knip / dependency-cruiser devDeps)
.cheshire/calm-layers.json                   (architecture test only)
tests/fitness/__fixtures__/calm-layers/...   (architecture test only)
```

Review against the Requirements:

- Is it a **test** under `tests/fitness/` that the existing CI already runs — *not* a new `fitness-functions.yml` workflow?
- Does it use the **native** tool from `repo-metadata.yml` — and is there **no `pmat`** in the runtime path?
- Does the architecture test actually **parse the BAR file**, or did the agent inline the rules? It should *read the BAR*, otherwise the next BAR re-implements everything.
- Is the **baseline floor honest** — the current measured value, not an aspirational zero that would make the test fail on merge?

Run locally before merging:

```bash
git checkout copilot/add-fitness-complexity
npm install
npm test -- fitness
```

You should see the Complexity test sitting right at the floor because of the planted `formatMovieResponse` function (cyclomatic 14), and the Dead Code test flag a stale export the seed code shipped with. The others should pass cleanly on the current movie-api.

### Step 5. Decide the ratchet (what to fix now vs. defer)

This is the moment that separates teams who *have* fitness functions from teams who *use* them. The Complexity floor is 14 because of the planted function. You have three honest options:

1. **Fix the high-CC function now**, with a follow-up issue and an agent run, then **lower the floor** (e.g. 14 → 10) so the gain is locked in. Same workflow as Part 3.
2. **Defer**, leaving the floor at 14 with an issue titled "Ratchet complexity 14 → 10 (refactor `formatMovieResponse`)" labelled `technical-debt`. The test stays green at the current floor; the issue stays open; the next person to touch that file ratchets it.
3. **Set a loose floor for a category you only want to *trend*** (e.g. duplication floor at the current 4% with a target of 3%) — track it without blocking day-to-day work on small wiggles.

In a live class, do option 1 for Complexity (fire one more agent run, then ratchet the floor) and option 2 for dead code (file an issue, defer). Demonstrate both. **The ratchet is the discipline: the floor only ever goes down.**

### Step 6. Merge and re-read the scorecard

When the fitness tests are green at their floors, merge. The disclosure footer names the packs the work touched (including the new calm-layer pack at v0.1.0) — see Part 6 for the signed manifest.

Refresh the Security Scorecard. Complexity, Technical Debt, and Test Coverage move; the repo climbs toward Supervised tier. Part 5 (the Caterpillar's Challenge) wires more scanners. Part 6 versions the packs. Part 7 makes the architecture rule *deterministic*.

## The architecture test in detail

The architecture fitness test deserves its own paragraph because it is the bridge into the rest of the workshop.

What it does, mechanically: runs dependency-cruiser over `src/`, maps each file to a layer via `.cheshire/calm-layers.json`, checks each import against the declared CALM flow, fails if a route imports from `db/` directly (skipping the service layer), or if `services/movie-service.ts` imports from `db/identity.ts` (cross-BAR data access), or any other layer-crossing not explicitly declared — against the baseline floor.

Why this matters for Part 7. The Red Queen's PreToolUse hook does the same check, but **before the agent's edit ever lands on disk**. The CI fitness test fires *after* the code is written. The Red Queen hook *blocks the write before it happens*. The test you generated today is the architectural rule; Part 7 turns the rule into a guardrail at agent-tool-call time. You need to *believe* the rule before you let it block your agent's tool calls.

## Q&amp;A

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 1. Why baselines instead of absolute thresholds?</summary>

Because an absolute threshold either fails on merge (if the seed code is worse than ideal) or has to be set so loose it never bites. The **baseline floor is the honest current state**: the test passes today, and it fails the moment the metric gets *worse*. Then you ratchet — every time you improve, lower the floor and lock the gain in.

This is how you change a real codebase without merging a 200-file refactor PR. You stop the bleeding immediately (no regressions past the floor), and you improve on a schedule (ratchet the floor toward the target as you touch each area). A gate that fails on day one teaches the team to disable it; a baseline that holds the line and tightens on your schedule earns trust.

</details>

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 2. The agent generated the architecture checker. How do I trust it is correct?</summary>

Three checks, in order.

1. **The fixture test.** The RCTRO required a synthetic violation fixture. Run it. If the fixture passes (no violation detected when one should be), the checker is broken; comment on the PR and re-dispatch.
2. **The audit on real code.** Run the test against the current `src/` and read the output. If it flags zero violations and you can see (by eye) a route file importing from `db/`, the layer mapping is wrong. Adjust `.cheshire/calm-layers.json` and re-run.
3. **The reverse audit.** Intentionally introduce a violation in a branch. Confirm the checker catches it. Revert.

This is how you validate any structural check, agent-generated or not. The fitness function for the fitness function is the fixture test plus the manual audit. One hour the first time buys years of trustworthy enforcement.

</details>

<details class="docs-details docs-card docs-card-muted">
<summary class="docs-details-summary">▶ Question 3. What stops the team from loosening a baseline instead of fixing the underlying issue?</summary>

Two things.

**The change goes through the same review.** Raising a floor is a PR. The PR needs a reason. The reason becomes part of the audit trail. "We raised the complexity floor because too many functions failed" is a much harder conversation than "we deferred one function and tightened the floor next sprint." The floor is *supposed* to only go down.

**The Red Queen, in Part 7.** Once Red Queen is installed, the policy reads the fitness outcomes and adjusts the tier. A repo whose baselines are loosened by override drops a tier. Lower tier means more PreToolUse restrictions on the agent. **Loosening gates makes the agent slower, not faster.** Wire incentives so the lazy path produces governed outcomes — do not rely on willpower.

</details>

## Try it yourself: a fifth fitness test

The team wants another gate: **no `any` types in production code** (`src/` excluding tests). Write the RCTRO for this. Pick the pack. Decide the baseline. Do not run Cheshire this time.

### What to write

1. **Role.** One sentence.
2. **Context.** Which files, which tool (`@typescript-eslint/no-explicit-any`), where the test lives (`tests/fitness/no-any.test.ts`).
3. **Task.** One sentence.
4. **Requirements.** Tool (native, never pmat), scope, exclusions, baseline floor = current count of `any`, ratchet target = 0.
5. **Output.** ESLint config, the fitness test, the baseline entry.

### Check your work

## What you learned

- **A fitness function is a structural assertion**, shipped here as a **committed test** under `tests/fitness/` that the existing CI runs — not a unit test, not a lint rule, and not a separate workflow.
- **The four structural categories:** Complexity (ESLint `complexity`), Duplicated code (`jscpd`), Dead code (`knip`), Architecture (`dependency-cruiser` + a CALM-layer rule that reads the BAR file). The tool per category is chosen on the Scaffold page and recorded in `repo-metadata.yml`.
- **Fast + native at runtime, always.** The fitness test runs a quick native tool. `pmat` is an authoring-time aid for designing the test locally — never a runtime dependency.
- **Baselines ratchet.** Each test asserts the metric is no worse than its floor; the floor only goes down. You stop regressions immediately and improve on a schedule, instead of merging a 200-file refactor or shipping a gate that fails on day one.
- **The Cheshire enrich → dispatch → review loop generalises.** Same workflow you ran in Part 3 on a security fix, applied here to build the gates themselves. *The pattern is the leverage.*
- **Your team co-authored its first prompt pack.** `.cheshire/prompts/maintainability/calm-layer-enforcement.md` did not exist before today.
- **The architecture test is the bridge to Part 7.** Today it runs in CI (after the agent wrote the code). In Part 7 the same rule becomes a PreToolUse hook (blocks the write before it lands). Same constraint, two enforcement points.
- **Validating an agent-generated checker is a fitness function for the fitness function.** Run the fixture, audit the real `src/`, do the reverse audit.
